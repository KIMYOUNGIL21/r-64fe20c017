3. 단계별 상세 — 실제 코드에서는 이렇게 돈다

③ 작사

무슨 일: 참조곡(예: 내사랑내곁에)의 정서·어법·구조의 '결'만 해부(가사 베끼기 금지) → N곡 초벌 → 스스로 검토(곡끼리 소재 겹침·상투어·채널 약속 위반) → 고쳐서 완성 → 복붙용 작업판.

코드에서는 ① 에이전트가 약속.md·채널기획서를 읽고 시작
② 완성본을 HTML로 생성 → <채널>\작사\<작업명>.html (곡마다 [가사 복사][스타일 복사] 버튼, 완료 체크는 브라우저에 자동 저장)
③ D:\work\lyrics-pages\<작업명>\index.html로 복사 → git push → 1~2분 뒤 kimyoungil21.github.io/r-64fe20c017/<작업명>/ 에 생김
④ 총괄PD가 그 URL을 카톡 나챗방으로 전송
수노 스타일 프롬프트에는 가수·곡명을 안 쓴다(수노가 차단). "1990s Korean soul ballad, raspy warm male vocal"처럼 묘사로 자동 변환.

④ 곡 생성 — 사장님 담당 구간

작업판에서 복사 → 수노에서 생성 → 마음에 드는 버전의 wav 다운로드. 이미지 3종 준비: 가로 클린(글자 없음, 영상 배경용) / 가로 제목본(유튜브 썸네일, 2MB 이하) / 세로 제목본(쇼츠 배경).

재료 넣는 곳 (파일 5종, 곡명으로 통일) 곡명.wav(음원) · 곡명.txt(가사) · 곡명-클린.png · 곡명.png · 곡명숏츠.png
→ 채널의 작업실\ 폴더에 넣고 "발매해줘" 하면 끝. wav→mp3 변환은 클로드가 알아서.

⑤ 제작: 타이밍 → 빌드

무슨 일: "가사 자막이 노래에 착착 맞게 뜨는" 비밀이 이 단계. 노래에서 목소리만 분리해 AI가 받아쓰기하고, 원래 가사와 글자를 맞춰 줄마다 시작 초를 계산한다. 그다음 영상 설계도를 만든다.

코드에서는 ① songs\<slug>.json (매니페스트) — 곡 정보가 전부 이 파일 하나에: 제목·파일명·색보정·쇼츠 컷 구간·예약 일시. 곡마다 새 스크립트 금지, 이 파일만 바꾼다 — 사고 방지 핵심 설계.
② pipe_setup.py — 재료를 작업폴더로 복사 + 가사에서 [Chorus] 태그 정리
③ pipe_timing.sh → demucs(보컬 분리) + whisper(받아쓰기) + 글자 정렬 → <slug>.timing.json (줄별 시작 초)
④ pipe_build.py → 롱폼 설계도 <slug>-mvB-fx.html (영상 설계도가 HTML 파일이다! 배경사진+자막+구독카드+효과가 코드로 박힘)
⑤ build_shorts3.py → 쇼츠 3컷: ffmpeg로 오디오 20초씩 자르고 컷별 설계도 3개 생성
쇼츠 3컷 규칙: ①1절 후렴 ②2절 후렴 ③클라이맥스 (2절 없으면 ①후렴 ②브릿지→파이널 진입 ③반전 펀치라인). 각 20초 강제 컷이라 펀치라인이 안 잘리게 뒤에서부터 역산해 컷 시작점을 잡는다.

⑥ 검수 게이트 1 — 제작 검수

무슨 일: 굽기 전에 자막 싱크와 쇼츠 규격을 검사. 여기 통과 못 하면 렌더 금지.

코드에서는 ① 검수.py — 구운 자막 시각 vs AI가 다시 들은 시각을 줄 단위 비교 (0.35초 이내 ✅)
② validate_short_fx.mjs — 쇼츠 표준 검사: 구독 CTA 위치, 길이, 상단 채널 타이틀
③ 검수가 ❌ 준 줄은 맹신 금지 — 분리한 보컬의 소리 크기 그래프(에너지 온셋)로 실측해서 최종 판정 (신나는 곡은 비트를 목소리로 착각하는 오탐이 잦다. 실제로 1호곡에서 22줄 지적 중 진짜 오류는 2줄뿐이었음)

⑦ 렌더

무슨 일: HTML 설계도를 mp4 영상으로 굽는다. 롱폼 3분짜리 기준 약 12분, 쇼츠 3컷까지 총 20~30분.

코드에서는 ① pipe_render.sh → hyperframes(HTML을 크롬으로 열어 녹화하듯 굽는 도구)로 렌더
② 결과: renders\곡명_가로16-9_MV.mp4 + 곡명_쇼츠1~3_MV_fx.mp4
③ 쇼츠 3개는 G:\내 드라이브\AI가수_숏폼\로 자동 복사 → 폰에서 바로 확인 가능
오래 걸리는 작업이라 별도 프로세스로 분리해 돌리고 로그만 지켜본다 (중간에 끊기는 사고 방지)

⑧ 발행 (업로드+예약)

무슨 일: 업로드 로봇이 채널 전용 크롬(로그인 저장됨)을 열고 유튜브 스튜디오를 사람처럼 눌러가며 업로드→썸네일→태그→예약까지. 화면에 크롬 창이 뜨는 게 이 로봇이다.

코드에서는 ① pipe_cfg.py — 업로드 설정 json 생성 (제목/설명/태그/썸네일 경로. AI 고지는 맨 아래 한 줄 자동 배치, 썸네일 2MB 초과면 여기서 정지)
② upload-one.js — 비공개 업로드 → 영상 ID 확보
③ schedule-video.js — 지정 날짜·시각으로 예약 (+ AI 합성콘텐츠 "예" 체크)
④ 쇼츠는 run_shorts3_upload.js — 3개 일괄 업로드+예약, 예약표에 완료 기록(재실행해도 중복 안 됨)
채널마다 크롬 프로필이 완전 분리(profile / profile-trot / profile-gayobang) — 채널이 섞이는 사고 원천 차단. 같은 프로필로 두 작업 동시 실행 절대 금지.

⑨ 검수 게이트 2 — QA 성적표

무슨 일: 제작에 관여하지 않은 검수반장이 약속.md를 채점표 삼아 [기계] 항목은 스크립트로, [판단] 항목(제목 후킹·컷 각도·가사 결)은 내용을 읽고 채점. 결과는 합격 / 조건부 합격 / 불합격.

코드에서는 ① qa-live.js — 유튜브 스튜디오 실물에서 제목·설명 첫줄/끝줄·시청자층·댓글 켜짐·공개상태를 읽어 json으로
② 업로드 설정 json(기대값) vs 실물 대조, 렌더 파일·G드라이브·예약표 확인
③ 성적표 형식 고정: ❌(즉시 조치) / ⚠️(주의) / ✅(통과 개수) / 검증 못 한 항목
실패 처리 정책: 기계적 문제(댓글 꺼짐·예약 오류·설명 누락)는 총괄PD가 즉시 수정 후 재검. 콘텐츠 판단(제목이 밋밋함 등)은 수정안과 함께 사장님 보고.

5. 철칙 — 사고에서 배운 것들

  • 영상 ID는 정확일치만. "최신 영상" 추측으로 잡으면 남의 영상을 재예약하는 사고가 난다 (실제 있었음).
  • 렌더 2개 동시 금지, 렌더 중 업로드 금지. 컴퓨터가 버벅이면 둘 다 망한다.
  • 설명란 AI 고지는 맨 아래 한 줄. 위에 있으면 감성 인트로가 죽는다. 업로더가 자동 교정.
  • 발행 후 댓글 켜짐 확인. 1호 업로드가 댓글 꺼진 채 나간 사고 → 이제 QA 필수 항목.
  • 검수 도구의 ❌는 실측으로 재판정. 신나는 곡에서 오탐 잦음.
  • QA 성적표 없이 "발매 완료" 보고 금지.
